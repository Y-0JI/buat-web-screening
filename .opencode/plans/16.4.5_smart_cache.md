# 16.4.5 — Smart Cache (TERPUSAT)

## Context & Prinsip (dari pesan user)
Smart Cache = **satu lapisan cache terpusat** untuk seluruh pengambilan data:
- Kurangi request berulang ke provider.
- TTL konsisten per jenis data.
- Dukung invalidasi & refresh.
- **Logika cache TIDAK tersebar di provider.**
- **Seluruh provider pakai mekanisme cache yang SAMA.**

  `news_repository` (1j), `stock_price_repository` (price 1j / verify 5m).

## Rancangan: CacheService terpusat
`backend/app/cache/service.py` (BARU) — pintu tunggal:
```python
class CacheService:
    def __init__(self, backend: CacheBackend):
        self._b = backend
    async def get(self, category, key): ...
    async def set(self, category, key, value, ttl=None): ...
    async def clear(self, category=None):   # None -> semua
```
- Backend tetap `MemoryCache` (sudah ada), tapi semua akses LEWAT `CacheService`.
- Konstanta TTL di `cache/ttl.py` (BARU): PROFILE/FUNDAMENTAL/NEWS/PRICE/VERIFY/
  SCREENER/SCREEN/TECHNICAL.
- Singleton `cache_service = CacheService(MemoryCache())` di `cache/__init__.py`
  supaya seluruh repo/provider/scheduler pakai instance & mekanisme SAMA.

2. **BARU `cache/service.py`** — `CacheService` (get/set/clear per kategori).

4. **`providers/idx_provider.py`** — hapus `_screener_cache/_screener_ts/_screener_lock`

5. **`scheduler.py`** — hapus `_screen_cache/_cache_ts` & TTL manual; pakai

6. **4 repository** (profile/fundamental/news/price) — arahkan ke `cache_service`

8. **BARU `routers/cache.py`** — `POST /api/cache/refresh?type=` → `cache_service.clear(type)`.
